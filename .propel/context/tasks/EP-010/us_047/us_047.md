# User Story - US_047

## Story ID
   * US_047

## Story Title
   * Inline Form Validation and Error Handling

## Description
  * As a user filling out forms, I want real-time inline validation with clear error messages, so that I can correct mistakes immediately without waiting to submit the entire form.

## Acceptance Criteria
  1. **Given** all forms in the application (login, appointment booking, intake, document upload, admin forms), **When** I interact with form fields, **Then** the system validates fields on blur (when I leave the field) for immediate feedback, displays validation errors inline below each field with red text + error icon, displays validation success with green checkmark icon for correctly filled fields (optional, use sparingly to avoid clutter), provides specific error messages instead of generic ones (e.g., "Email must include @" instead of "Invalid email"), validates common field types: Email (RFC 5322 compliant regex), Phone (US format (XXX) XXX-XXXX or international +X XXX XXX XXXX), Date (valid date range, no future dates for DOB, no past dates for appointments), Required fields (not empty or whitespace only), Custom validations (e.g., insurance member ID format per provider), displays character count for limited fields (e.g., "Reason for visit: 45/200 characters"), implements form-level validation summary at top showing all errors with links to fields (e.g., "3 errors found: Email, Phone, Date of Birth"), prevents form submission if validation fails (submit button disabled with tooltip "Please fix 3 errors"), preserves form data during validation errors (no data loss on validation failure), implements debounced validation for expensive checks (e.g., username availability check after 500ms of no typing), provides ARIA live regions for error announcements to screen readers (aria-live="assertive" for critical errors), displays loading spinners for async validations (e.g., checking insurance eligibility), allows users to toggle password visibility with eye icon for password fields, and maintains consistent error styling across all forms using design tokens (--color-error for text, --color-error-light for field border).

## Edge Cases
   * What happens when validation rules change during session? (Re-validate all fields on rule update, show notification "Validation rules updated")
   * How are network errors during async validation handled? (Show "Unable to validate - check connection" message, allow user to proceed with warning)
   * What if user pastes invalid data? (Validation triggers on paste event immediately, shows error before blur)

> **Wireframe Details (Form Validation Patterns):**
> - **Default State**: Input field with label, placeholder text (lighter gray #999), border color `--color-border` (#E0E0E0), no icons
> - **Focus State**: Input field with blue border `--color-primary-main` (2px), label moves up or changes color, no error message yet
> - **Error State** (triggered on blur if invalid):
>   - Input border changes to `--color-error` (#C62828, 2px solid)
>   - Error icon (red ⚠️ or ✗) appears at right edge of input field
>   - Error message appears below field in red text (#C62828, 14px font-size, icon + text)
>   - Example: "⚠️ Email must include @ symbol"
>   - Field background may have light red tint #FFEBEE for additional visibility
>   - ARIA attributes: aria-invalid="true", aria-describedby="email-error-id"
> - **Success State** (optional, use for critical fields only):
>   - Input border changes to `--color-success` (#2E7D32)
>   - Green checkmark icon (✓) appears at right edge
>   - No message needed (success is implied)
> - **Loading State** (async validation):
>   - Spinner icon at right edge of input
>   - Border color blue (validating)
>   - Helper text "Checking..." below field in gray
> - **Form-Level Error Summary** (top of form):
>   - Red banner with icon + text: "⚠️ Please fix 3 errors to continue:"
>   - Bulleted list of errors with clickable links: "- Email address is invalid" (clicking focuses email field)
>   - Dismissible with X button, reappears if user tries to submit with errors
> - **Submit Button States**:
>   - Disabled (errors present): Gray background, cursor not-allowed, tooltip on hover "Fix 3 errors to submit"
>   - Enabled (no errors): Primary button style, cursor pointer
>   - Loading (submitting): Spinner icon + "Submitting..." text, disabled to prevent double-submit
> - **Specific Validations**:
>   - Email: Regex `/^[^\s@]+@[^\s@]+\.[^\s@]+$/`, error "Email must include @ and domain (e.g., user@example.com)"
>   - Phone: Format (XXX) XXX-XXXX, auto-formats as user types, error "Phone must be 10 digits"
>   - Date of Birth: Must be past date, age ≥18 years, error "Patient must be 18 or older" or "Date cannot be in the future"
>   - Password: Min 8 characters, 1 uppercase, 1 number, 1 special char, strength meter (Weak/Medium/Strong) below field
>   - Required fields: Asterisk (*) in label, error "This field is required"
>   - Character limits: Counter below field "45/200", turns red at 190+ characters as warning
> - **Validation Timing**:
>   - On Blur: Triggered when user leaves field (most common)
>   - On Change: For fields needing immediate feedback (password strength, character count)
>   - On Submit: Final validation of entire form before API call
>   - Debounced: Expensive async checks (username availability, insurance eligibility) wait 500ms after user stops typing
