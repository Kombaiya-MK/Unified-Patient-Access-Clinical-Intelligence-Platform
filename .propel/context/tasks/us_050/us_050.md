# User Story - US_050

## Story ID
   * US_050

## Story Title
   * Zero-Downtime Deployment with PM2 Cluster Mode

## Description
  * As a DevOps engineer, I want to deploy application updates without service interruption using PM2 cluster mode and rolling restarts, so that users experience zero downtime during releases.

## Acceptance Criteria
  1. **Given** the Node.js backend is deployed using PM2 process manager, **When** I deploy a new version, **Then** the system runs PM2 in cluster mode with number of instances equal to CPU cores (e.g., 4 cores = 4 instances) per NFR-REL05, implements zero-downtime deployment using `pm2 reload` command which gracefully restarts instances one at a time (rolling restart): sends SIGINT to instance 1 → waits for graceful shutdown (closes keep-alive connections, finishes in-flight requests) → starts new version instance 1 → waits for health check success (HTTP 200 on /health) → repeats for instances 2, 3, 4, ensures at least N-1 instances running during reload (if 4 instances, minimum 3 always serving requests), implements graceful shutdown handler in Node.js: listens for SIGINT signal, stops accepting new connections, finishes processing in-flight requests with 30s timeout, closes database connections gracefully, exits with code 0, configures PM2 ecosystem.config.js with: name "upaci-backend", script "server.js", instances "max" (all CPU cores), exec_mode "cluster", kill_timeout 30000 (30s for graceful shutdown), max_memory_restart "1G" (restart if memory exceeds 1GB), env variables (NODE_ENV=production, DB_HOST, REDIS_HOST), implements health check endpoint GET /health responding: 200 OK if database connected + Redis connected + AI service reachable, 503 Service Unavailable if any dependency down, configures PM2 for automatic restart on crash with max_restarts 10, min_uptime 60000 (consider stable after 60s), implements deployment pipeline: build new version → run tests → deploy to staging → smoke tests → deploy to production with `pm2 reload upaci-backend`, logs all PM2 events (restart, reload, crash) to centralized logging (PM2 logs + Prometheus exporter), monitors deployment success with metrics: deployment_duration_seconds histogram, instances_running gauge, zero_downtime_achieved counter, and documents deployment process in .propel/docs/zero-downtime-deployment.md with rollback procedures (git revert + redeploy or pm2 reload with previous version).

## Edge Cases
   * What happens when new version fails health check? (PM2 keeps old version running, alerts admin, deployment marked as failed)
   * How are database migrations handled during zero-downtime deploy? (Backward-compatible migrations run before code deploy, breaking changes require maintenance window)
   * What if all instances crash simultaneously? (PM2 automatically restarts all instances, circuit breaker opens to protect database from stampede)

> **Implementation Notes:**
> - **PM2 Ecosystem Configuration** (ecosystem.config.js):
>   ```javascript
>   module.exports = {
>     apps: [{
>       name: 'upaci-backend',
>       script: './server.js',
>       instances: 'max', // Use all CPU cores
>       exec_mode: 'cluster',
>       max_memory_restart: '1G',
>       kill_timeout: 30000, // 30s graceful shutdown
>       wait_ready: true, // Wait for app.listen() before considering ready
>       listen_timeout: 10000, // 10s timeout for app to listen
>       max_restarts: 10,
>       min_uptime: '60s', // Stable after 60s
>       env: {
>         NODE_ENV: 'production',
>         PORT: 3000,
>         DB_HOST: 'localhost',
>         DB_PORT: 5432,
>         REDIS_HOST: 'localhost',
>         REDIS_PORT: 6379
>       },
>       error_file: './logs/pm2-error.log',
>       out_file: './logs/pm2-out.log',
>       log_date_format: 'YYYY-MM-DD HH:mm:ss Z',
>       merge_logs: true
>     }]
>   };
>   ```
> - **Graceful Shutdown Handler** (server.js):
>   ```javascript
>   const server = app.listen(PORT, () => {
>     process.send('ready'); // Signal PM2 app is ready
>     console.log(`Server running on port ${PORT}`);
>   });
>   
>   process.on('SIGINT', async () => {
>     console.log('SIGINT received, starting graceful shutdown...');
>     server.close(async () => {
>       console.log('HTTP server closed');
>       await db.close(); // Close PostgreSQL connections
>       await redis.quit(); // Close Redis connections
>       console.log('All connections closed');
>       process.exit(0);
>     });
>     
>     // Force shutdown after 30s timeout
>     setTimeout(() => {
>       console.error('Forced shutdown after 30s timeout');
>       process.exit(1);
>     }, 30000);
>   });
>   ```
> - **Deployment Commands**:
>   - Initial start: `pm2 start ecosystem.config.js`
>   - Zero-downtime reload: `pm2 reload upaci-backend` (rolling restart)
>   - Scaling: `pm2 scale upaci-backend 8` (change instance count)
>   - Monitoring: `pm2 monit` (real-time dashboard), `pm2 logs upaci-backend` (view logs)
>   - Save configuration: `pm2 save` (save current process list for auto-restart on reboot)
>   - Startup script: `pm2 startup` (generate OS startup script for PM2 daemon)
> - **Health Check Endpoint** (/health):
>   ```javascript
>   app.get('/health', async (req, res) => {
>     try {
>       await db.query('SELECT 1'); // Test database
>       await redis.ping(); // Test Redis
>       // Optional: Test OpenAI API with timeout
>       res.status(200).json({ status: 'healthy', timestamp: new Date() });
>     } catch (err) {
>       res.status(503).json({ status: 'unhealthy', error: err.message });
>     }
>   });
>   ```
> - **Deployment Pipeline** (CI/CD):
>   1. Git push triggers pipeline
>   2. Run unit + integration tests
>   3. Build artifacts (npm run build)
>   4. Deploy to staging: `ssh staging "cd /app && git pull && npm install && pm2 reload upaci-backend"`
>   5. Run smoke tests on staging (curl health endpoint, test critical flows)
>   6. Deploy to production: `ssh prod "cd /app && git pull && npm install && pm2 reload upaci-backend"`
>   7. Monitor production metrics for 10 minutes (error rate, response time)
>   8. Rollback if error rate spikes: `ssh prod "cd /app && git checkout previous_tag && pm2 reload upaci-backend"`
> - **Monitoring**: PM2 Plus (SaaS monitoring), custom Prometheus exporter (pm2-prometheus-exporter), Grafana dashboards showing instance count, CPU/memory per instance, deployment timeline
