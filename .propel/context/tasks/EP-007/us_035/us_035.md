# User Story - US-035

## Story ID
   * US_035

## Story Title
   * Admin User Management Interface

## Description
  * As an admin, I want to create, update, and deactivate user accounts with role assignment through an intuitive interface, so that I can efficiently manage platform access for patients, staff, and administrators.

## Acceptance Criteria
  1. **Given** I am logged in as admin, **When** I access the User Management page (SCR-013), **Then** I see a table with columns: Email, Role (Patient/Staff/Admin), Department, Status (Active/Inactive), Last Login, and Actions (Edit/Deactivate).

  2. **Given** I click "Create User", **When** I fill the modal form with email, password, role selection, and department dropdown (visible only for Patient role per FR-022), and click "Save", **Then** the system creates the user account, sends email verification, and displays success message "User created successfully".

  3. **Given** I select a user and click "Edit", **When** I modify the role or department and save, **Then** the system updates the user record, logs the change to audit log, and if role changed, invalidates any active sessions for that user to force re-authentication.

  4. **Given** I click "Deactivate" next to a user, **When** I confirm the action, **Then** the system sets active=false, invalidates all user sessions, logs the deactivation to audit log, and the user cannot log in (receives "Account deactivated" message).

## Edge Cases
   * What happens when admin tries to create user with duplicate email? (Display error: "User with this email already exists")
   * How does the system prevent admin from deactivating their own account? (Disable "Deactivate" button for current admin user, show tooltip "Cannot deactivate your own account")
   * What if admin assigns Patient role but doesn't select department? (Display validation error: "Department is required for Patient role")
   * How are password complexity requirements enforced during user creation? (Frontend + backend validation: min 8 chars, 1 uppercase, 1 number, 1 special char)

> **Wireframe Details (SCR-013):**
> - **User table**: Sortable by email/role/status, search bar (filters by email/name), pagination (20 per page)
> - **Create User button**: Top-right, primary button, opens modal
> - **Create User modal**: Fields - Email (email validation), Password (show/hide toggle, strength indicator), Confirm Password, Role (dropdown: Patient/Staff/Admin), Department (dropdown, conditionally visible only for Patient role), Submit + Cancel buttons
> - **Edit User modal**: Same as create but password optional (leave blank to keep existing), shows current values pre-filled
> - **Deactivate confirmation**: "Are you sure you want to deactivate [email]? They will no longer be able to log in." with Cancel + Deactivate buttons
> - **Status badges**: Green (Active), Gray (Inactive)
> - **Action buttons**: Edit icon (pencil), Deactivate icon (user-x), disabled/grayed for self-deactivation
> - **Responsive design**: Mobile (cards with expand details), tablet/desktop (table layout)
> - **Accessibility**: ARIA labels for all buttons/inputs, keyboard navigation (Tab through table, Enter to edit), focus indicators
