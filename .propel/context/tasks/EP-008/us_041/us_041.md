# User Story - US_041

## Story ID
   * US_041

## Story Title
   * Circuit Breaker for AI Service Resilience

## Description
  * As a backend developer, I want to implement circuit breaker pattern for OpenAI API calls, so that the system gracefully degrades when AI services are unavailable instead of cascading failures.

## Acceptance Criteria
  1. **Given** the system makes calls to OpenAI API for AI intake, document extraction, medical coding, and conflict detection, **When** OpenAI API failures exceed 50% error rate within 1-minute rolling window (measured per endpoint), **Then** the circuit breaker transitions to "Open" state and blocks subsequent API calls for 60 seconds cooldown period, returns graceful fallback responses to users: AI intake → fallback to manual form-only mode (SCR-007 form without chat), Document extraction → queue document for later processing with "Processing Delayed" status, Medical coding → show "AI suggestion unavailable - use manual coding" message, Medication conflicts → use rule-based local validation only (basic drug interaction database), displays circuit breaker status in admin dashboard (SCR-004) with indicator (Closed=Green, Half-Open=Yellow, Open=Red), logs all state transitions to audit log with timestamp and error details, attempts Half-Open state after cooldown by sending single test request (health check endpoint), transitions back to Closed if test succeeds or back to Open if test fails (exponential backoff 60s → 120s → 300s), notifies admin via email/SMS when circuit opens (critical alert), implements circuit breaker per OpenAI model endpoint (GPT-4 for intake, GPT-4V for extraction, separate breakers), and tracks metrics: circuit_breaker_state gauge (0=closed, 1=half-open, 2=open), api_failure_rate histogram, fallback_activation_count counter.

## Edge Cases
   * What happens if circuit opens during activeappointment booking? (User can book but AI features disabled with "Limited functionality" banner)
   * How are queued documents processed after recovery? (Background job retries failed extractions in FIFO order with rate limiting 10/minute)
   * What if fallback logic also fails? (System returns 503 Service Unavailable with retry-after header, logs critical error for manual intervention)

> **Implementation Notes:**
> - **Circuit Breaker Library**: Use `opossum` (Node.js circuit breaker library) or custom implementation with three states: Closed (normal), Open (blocking calls), Half-Open (testing recovery)
> - **Configuration**: Failure threshold 50%, request volume threshold 10 (minimum requests before opening), timeout 30s per request, rolling window 60s, cooldown period 60s
> - **State Machine**: Closed → (>50% failures in 1min) → Open → (after 60s) → Half-Open → (test success) → Closed OR (test fail) → Open
> - **Graceful Degradation**: Each AI feature has defined fallback: Intake=manual form, Extraction=queue+retry, Coding=show manual codes, Conflicts=basic rules
> - **Monitoring**: Prometheus metrics exported, Grafana dashboard showing state per endpoint, PagerDuty alert on state Open, health check logs for debugging
> - **Testing**: Unit tests for state transitions, integration tests with mock OpenAI failure scenarios, load test with circuit breaker activation
