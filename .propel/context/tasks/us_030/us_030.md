# User Story - US_030

## Story ID
   * US_030

## Story Title
   * Multi-Document Deduplication Logic

## Description
  * As a clinical staff member, I want the system to automatically detect and merge duplicate information across multiple uploaded documents, so that patient profiles show unified data without redundant entries.

## Acceptance Criteria
  1. **Given** multiple documents have been processed for the same patient with extracted data, **When** the deduplication service runs (triggered after each document extraction), **Then** the system compares extracted fields across documents using similarity algorithms (fuzzy matching for names with 85% threshold, exact matching for structured data like medication names, date range overlap for lab results from same test within 7 days), identifies duplicates with >95% accuracy per AIR-Q02, merges duplicate entries by keeping mostrecent value or highest confidence value, flags conflicting values (e.g., different allergy lists from two documents) with status="Conflict" for staff review, creates merged PatientProfile record, and logs all merge decisions to audit log showing source documents and merge rationale.

## Edge Cases
   * What happens when two documents have contradictory information? (Don't auto-merge, flag both as conflicts, require staff to manually resolve in US-034)
   * How are medication dosage changes over time handled? (Keep all as timeline entries with document_date, don't treat as duplicates if dates differ by >30 days)
   * What if deduplication algorithm has low confidence (< 85%)? (Skip auto-merge, present side-by-side comparison to staff for manual decision)
